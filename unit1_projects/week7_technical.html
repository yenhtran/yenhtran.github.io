SQL Injections

What is a SQL Injection?

SQL injections is a type of web attacking technique that is used by hackers to steal information and data from organization. The way a SQL injection works is when an attacker passes SQL commands (statements) into the user input fields and is able to gain access and/or even wipe out an organization's database directly. Websites that have features that allow legitimate users to input data (i.e. login pages, request forms, feedback forms, shopping carts, etc.) are vulnerable to these attacks.

Let's say you have a simple login page that any legitimate user would enter their username and password to enter the site. When valid credentials are submitted and verified, the user is allowed access. However, if someone wanted to come in add maliciously effect your database, they could use a SQL injection.

A SQL Injection Example:

Typically, the way one talks to a database by issuing a command like this: 


	SELECT * FROM Users
		WHERE Username = "Jane"
This command will have the database send back everything it knows about "Jane". But the catch are those quotation marks ("") becuase if we not carefull about what we send, we can cause some problems like the one below. Lets say I submit a webform with: 

	SELECT * FROM Users
		WHERE Username = Jane"
Notice that lone quotation mark at the end? What will happen is that the program will automatically add a set of quotation marks to the datebase like so: "Jane"". Unfortunately this will confuse the program, which in turn will return an error. The catch is you can cause a lot of problems this way. This is because SQL has many other commands to it including: INSERT (to add new data), UPDATE (to change data), and DELECT (to remove data). So if a hacker submited the following: 

	SELECT * FROM Users
		WHERE Username = Jane"; DROP ALL DATABASES
What will happen is that SQL thinks what is behind the semicolon is a newcommand and will run said command and in this case, DELETE EVERYTHING. (And yes, at that point you may need to dust off that resume.)

How does one combat this? PREPARED STATEMENTS

Prepared statements is the most common way to combat this problem and all programmer should be including these in their code. What essentially it is, it's a hack. It's a hack on top of a hack, but it does keep the database safe. What it looks like is as follows: 


	SELECT * FROM Users
		WHERE Username = ?
In a extremely simplified nutshell, what this is telling the program that the ? is data to be inserted in, not a command. No matter what the data looks like, it should be stored in the database and is not a command to be run. (Programmers should be sure the check the current security guidelines to write this.)
